# Material component contract

Run inside the `material-component` orchestrator context. Receive the current invocation, preflight, orientation, concern plan, actual dependency inventory, target/audit results, owners, exports, consumers, legacy state, and proof.

Block when invocation scope is missing, current code was not inspected, a required lane is missing, dependency ownership/readiness is unknown, or production work preceded the gate.

## Responsibility

Synthesize:

- applicable target/source decisions;
- current ownership and canonicalization state;
- complete recursive dependency closure and exact prerequisites;
- durable public, token, style, and motion facts;
- highest-priority complete correction;
- proof lane, failing observation, compatibility/visual impact, operator requirement, and completion condition;
- remaining required gaps and exact external blockers.

Do not repeat accepted research. Challenge contradictions, missing imports/owners, unsupported generalizations, missing proof, and relocation presented as readiness.

## Scope and dependencies

A family-only request is `full-family`. README, roadmap, or a previous bounded unit cannot change it to `focused-correction`.

For every required dependency record:

- owning Material workflow and canonical owner;
- classification and recursive dependencies;
- implementation, token, semantics/lifecycle, and public-contract readiness;
- all direct consumers of a changed public contract or extension;
- legacy owner disposition and compatibility route;
- proof, independent review, replacement obligation, and orchestration status.

A used dependency remains in the caller's orchestration. A temporary legacy or missing family-agnostic contract becomes an `execute-now` `material-foundation` prerequisite. Another official component becomes an `execute-now` owning `material-component` prerequisite. Each prerequisite must canonicalize its exact required contract to the same readiness standard as a directly requested artifact, including nested prerequisites, then return automatically.

Moving files, adding forwarding/barrels, migrating imports, or passing path guards cannot mark a dependency ready. Known defects, legacy-owned token declarations, incompatible consumers, or missing owner review keep it blocked.

Creating/preserving a canonical owner, changing the root export, migrating consumers, removing/forwarding the legacy owner, or claiming readiness widens closure to the complete supported surface. Premature canonicalization makes dependency closure or safe rollback the highest-priority correction.

## Documentation

Update an owner README only when the correction changes its durable contract. Never write invocation scope, current stage, correction state, backlog, review result, shell output, or future passes there.

Return current correction and continuation state only through the structured result below.

## Exit gate

Pass only when current code and scope are valid, required lanes are complete, actual dependencies have explicit canonical owner/readiness/orchestration status, all changed public contracts have direct-consumer compatibility obligations, canonicalization widened closure correctly, the correction is highest priority, prerequisites are exact, and proof/compatibility decisions are locked.

## Forbidden

- direct user invocation or production edits;
- trusting README execution state;
- treating used dependencies as outside orchestration;
- accepting relocation, forwarding, barrels, migrated imports, or green guards as readiness;
- accepting open recursive closure for canonicalization/adoption;
- choosing lower-priority work around ownership blockers;
- describing prerequisites as future operator work;
- writing execution history, backlog, status blocks, registries, checklists, scorecards, or ledgers;
- roadmap advancement or Git operations.
