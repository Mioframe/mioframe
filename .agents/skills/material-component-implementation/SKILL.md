# Material component implementation

Internal stage only. Use it after the current correction objective is ready and every required foundation prerequisite is available.

## Inputs

Read:

- root and applicable nested `AGENTS.md` files;
- `src/shared/ui/material/docs/architecture.md`;
- the implementation phase of `src/shared/ui/material/docs/component-development.md`;
- the current family README, including canonical target, alignment map, correction units, proof classification, and required foundation contracts;
- current production owners, affected representative consumers, public exports, stories, tests, and relevant verification mappings.

Load Vue and testing skills only for the exact implementation or proof layer currently required.

## Responsibility

Execute each documented correction unit in order:

```text
faithful failing or prepared proof
→ smallest owner correction or local owner replacement
→ focused verification
→ affected browser and representative-consumer validation
→ alignment-map update
```

Before production edits, create the smallest faithful proof for the canonical expected behavior and confirm it fails for the expected reason when a pre-code executable proof is applicable. Define browser scenarios before implementation when real browser semantics own the contract. Do not create visual baselines before official comparison and acceptance.

Implement only the owners required by the current correction unit. Preserve owners already classified `confirmed-compliant` or `project-extension` unless new evidence invalidates that classification.

Keep public Vue components as thin composition roots. Use owner-local TypeScript modules, focused composables, and explicit owner-local style or motion modules when responsibilities, reasons to change, or proof owners differ. Do not split files mechanically and do not add wrapper components or DOM nodes merely for separation.

Choose correction versus replacement at the smallest owner boundary:

- correct incrementally when ownership is sound and the gap is local;
- replace one owner when its contract is predominantly wrong or incremental repair would add more workaround logic;
- never rewrite the whole family merely to make the implementation appear new.

Do not expand beyond the current objective before its units are coherent. Validate every materially affected real consumer category before reporting the objective complete.

When implementation evidence invalidates the canonical target, classification, ownership, decomposition, compatibility, proof, or foundation decision, return an exact contract blocker. Preserve completed unaffected units. Do not add a consumer workaround or silently rewrite the README.

## Exit gate

Pass only when:

- every current correction unit meets its documented completion condition;
- applicable failing proof now passes and existing proof has been reclassified honestly;
- implementation matches documented owners and order;
- public composition roots remain focused;
- style and motion ownership are explicit and actual rendered-property routes match them;
- affected representative consumers work without contract workarounds;
- no required foundation gap remains;
- the alignment map is updated from evidence;
- the repository is complete and independently valid even when the family remains `converging`.

## Forbidden

- direct user invocation;
- production edits before the current correction-unit proof gate;
- silent family-contract or alignment-classification changes;
- preserving code only because it is legacy or replacing code only to reduce similarity with legacy;
- tests that protect implementation details or known legacy defects as canonical behavior;
- monolithic implementation that ignores documented independent owners;
- file fragmentation that only moves lines without clarifying ownership or proof;
- migration of remaining consumers or complete legacy-owner removal unless explicitly delegated through adoption;
- roadmap updates or starting another stage;
- speculative surface, abstractions, managers, registries, validators, aliases, or extension points;
- unnecessary DOM nodes or generic tests for behavior the project does not own;
- deleting or rewriting the entire family when smaller complete owner corrections are sufficient.
