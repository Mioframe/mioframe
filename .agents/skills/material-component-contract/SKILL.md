# Material component contract

Run inside the `material-component` orchestrator context. Receive the current invocation, preflight, orientation, concern plan, actual dependency inventory, target/audit results, owners, exports, consumers, legacy state, and proof.

Block when invocation scope is missing, current code was not inspected, a required lane is missing, dependency ownership is unknown, or production work preceded the gate.

## Responsibility

Synthesize:

- applicable target/source decisions;
- current ownership and canonicalization state;
- complete dependency closure and exact prerequisites;
- durable public, token, style, and motion facts;
- highest-priority complete correction;
- proof lane, failing observation, compatibility/visual impact, operator requirement, and completion condition;
- remaining required gaps and exact external blockers.

Do not repeat accepted research. Challenge contradictions, missing imports/owners, unsupported generalizations, and missing proof.

## Scope and dependencies

A family-only request is `full-family`. README, roadmap, or a previous bounded unit cannot change it to `focused-correction`.

For every required dependency record owner, classification, readiness, proof, replacement obligation, and orchestration status.

A used dependency remains in the caller's orchestration. A temporary legacy or missing family-agnostic contract becomes an `execute-now` `material-foundation` prerequisite. Another official component requires its owning family public contract. Neither is terminal or out of scope.

Creating/preserving a canonical owner, changing the root export, migrating consumers, removing/forwarding the legacy owner, or claiming readiness widens closure to the complete supported surface. Premature canonicalization makes dependency closure or safe rollback the highest-priority correction.

## Documentation

Update an owner README only when the correction changes its durable contract. Never write invocation scope, current stage, correction state, backlog, review result, shell output, or future passes there.

Return current correction and continuation state only through the structured result below.

## Exit gate

Pass only when current code and scope are valid, required lanes are complete, actual dependencies have explicit owner/readiness/orchestration status, canonicalization widened closure correctly, the correction is highest priority, prerequisites are exact, and proof/compatibility decisions are locked.

## Forbidden

- direct user invocation or production edits;
- trusting README execution state;
- treating used dependencies as outside orchestration;
- accepting open closure for canonicalization/adoption;
- choosing lower-priority work around ownership blockers;
- describing prerequisites as future operator work;
- writing execution history, backlog, status blocks, registries, checklists, scorecards, or ledgers;
- roadmap advancement or Git operations.
