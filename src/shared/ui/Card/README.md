# Card Contract

`MDCard` is a Material Cards **container/surface component**. The only Material card element `MDCard` implements is the container itself: variants, card component tokens, surface context, interaction model, state layer, ripple, focus, disabled, and dragged behavior. Everything Material's Cards docs show _inside_ a card — images, headline/subhead/supporting text, buttons, lists, selection controls, linked text, overflow menus — is example content, not a required `MDCard` API surface. Consumers compose that content themselves inside the default slot using their own semantic tags and other shared UI primitives; `MDCard` is aligned with Material Cards as a container without owning any of it.

- `MDCard` implements the three Material card variants: `elevated`, `filled`, or `outlined` (`variant`, default `filled`).
- `mode` picks one of the two Material-documented interaction shapes, never both at once:
  - `static` (default): a non-actionable container for rich content, including internal buttons/links that own their own actions. No role, no tabindex, no hover state layer, no ripple.
  - `button` / `link`: a directly actionable card surface. The whole card is the native `button`/`a[href]`, with `MDStateLayer` + ripple for hover/focus/pressed/dragged, and an `action` emit. Material treats an actionable card as the whole tappable surface, so it must not contain its own nested actionable controls — use `mode="static"` for cards that need internal buttons/links.
- `mode="button"` renders the whole card as a native `<button>`, which restricts content to safe phrasing content (text, inline elements, icons) per the native `<button>` content model. Block-level layout and interactive descendants are invalid inside it. MDCard does not restrict slot content at the type level (the default slot is generic across all modes), but it warns in development (`cardDevWarnings.ts`) when a `mode="button"` card's rendered content includes a tag outside the phrasing-content allow list. Treat the warning as a guard for simple button-card content, not as the source of the nested-action rule.
- `mode="link"` renders the whole card as a native `<a>`, which accepts flow content, so it has no equivalent native `<button>` phrasing-content restriction. Nested interactive descendants (buttons, links) remain an unsupported combination for `mode="link"` too — not because `<a>` has the same content model as `<button>`, but because Material's accessibility guidance avoids stacking actionable surfaces. MDCard does not enforce this with a pre-render scan of the default slot (the slot renders arbitrary consumer content, so a reliable scan is not a narrow, low-risk check); treat it as a documented unsupported pattern rather than an enforced restriction.
- `disabled` only has meaning for actionable modes. A disabled `button` card uses native `disabled`. A disabled `link` card omits the `href` attribute entirely (so the anchor has no native navigation affordance for middle-click, context menu, or "open in new tab") and keeps `aria-disabled="true"` + `tabindex="-1"` for assistive tech, since links ignore the native `disabled` attribute. `href` is restored automatically once `disabled` is cleared.
- `dragged` is an externally controlled prop (e.g. driven by a sortable/reorder consumer), not native HTML drag detection.
- `MDCard` has no headline/supporting-text slots or `title`/`description` props, and does not need them for Material Cards alignment: Material documents those as example card content, not required container API. Consumers keep their own semantic tags (`h1`-`h3` for headings, `p` for supporting text, `span` for inline text) and apply the global Material typography utility classes (`.md-typescale-*` in `shared/lib/md/typography.css`) for type scale. Color, layout, spacing, and copy stay owned by the widget/feature composing the card.

## Material surface context

- MDCard is a Material surface owner: its root maps the resolved card container color to `--md-container-color`, sets `--md-content-color` to the card's content role, and derives `--md-current-container-color`/`--md-current-content-color` from those two so nested Material primitives inherit the card's surface, not the pane/page surface underneath it.
- Card has no documented `content.color` token (the official `md.comp.*-card` token graph defines container, disabled, focus-indicator, and hover/focus/pressed/dragged state-layer tokens, but no content-color token), so `--md-content-color` defaults to `--md-sys-color-on-surface` for every variant.
- The card's actionable ripple (`useRipple`/`ripple.css`) falls back to `--md-content-color` when no `--md-ripple-color` is set. Because on-surface is also the documented `*.pressed.state-layer.color` for all three card variants, this generic fallback already resolves to the correct pressed state-layer color — no card-specific ripple override was added to `shared/ui/State`.
